# brutal_mlp API Contracts

This file is the public contract reference for `include/brutal_mlp/brutal_mlp.hpp`.
When overloads share the same contract, they are grouped in one row and every covered
signature is named.

## Global Rules

- Sizes use `std::size_t`; dimensions must be positive unless the function explicitly accepts an empty object.
- Public validation APIs throw `std::invalid_argument` for bad user input and `std::runtime_error` for I/O, unsupported serialized data, corruption, or unavailable runtime state.
- Inference hot-path methods marked `noexcept` never throw; they report `InferenceStatus` or assume the caller has already satisfied the contract.
- Every scalar supplied by the user must be finite unless the contract explicitly allows a non-finite value for diagnostics.
- `Matrix` is a vector of rows; all rows must have the expected width. `DenseMatrix` is row-major contiguous storage.
- Non-owning dataset views require caller-owned data and callback contexts to outlive the object and any training call using it.
- `TrainingModel` mutates during `fit`, `set_parameters`, load/resume, and optimizer updates; do not share one mutable instance across threads without external synchronization.
- `InferenceModel` and `CompiledModel` are immutable after construction and can be shared across threads. Scratch/workspace buffers are caller-owned and must not be shared by concurrent calls unless externally synchronized.
- API stability labels:
  - Stable: intended to remain source-compatible within the current major version.
  - Compatibility: kept for existing users; new code should prefer the named replacement.
  - Experimental: supported and tested, but details may evolve before a major-version guarantee.

## Recommended Hardening Order

1. Gradient checking, normalization, and metrics.
2. Allocation-free `InferenceModel` / `CompiledModel`.
3. Clean float/double split and contiguous storage.
4. Streaming datasets, checkpoints, and training diagnostics.
5. Versioned binary format and frozen model validation.
6. Performance optimization, multithreading, and representative benchmarks.
